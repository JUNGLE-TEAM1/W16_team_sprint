# 개발 순서 문서

## 운영 문서 정렬 원칙

이 문서는 현재 프로젝트의 스프린트 번호와 Decision Harness 기록의 기준 문서다.

회의 중 새로 정리한 `docs/ai_knowledge_board_sprint_plan.md`는 운영 보강 문서로 사용한다. 단, 스프린트 번호는 이 문서의 기존 체계를 유지한다.

- 이 문서의 Sprint 1은 인증 구현이다.
- `docs/ai_knowledge_board_sprint_plan.md`의 Sprint 1에 해당하는 FastAPI, React, PostgreSQL 기반 설계 항목은 별도 Sprint 1로 진행하지 않는다.
- 해당 기반 설계 항목은 Sprint 0의 기반 조건 또는 Sprint 1 진행 전/진행 중 확인해야 하는 선행 조건으로 흡수한다.
- 이미 작성된 `docs/decisions/sprint-1/` Decision 기록은 기존 체계대로 Sprint 1 인증 구현 기록으로 유지한다.
- 인증 방식은 이 문서의 기존 결정에 따라 JWT를 기본 기준으로 유지한다. Session은 참고 또는 추후 확장 학습 항목으로 둔다.

새 회의 문서에서 보강해 반영할 내용은 다음이다.

- 각 스프린트는 개념 학습만이 아니라 화면 또는 API에서 실제 동작하는 구현 결과를 포함한다.
- 각 스프린트 완료 시 핵심 요청 흐름, DB 저장 내용, 실패 케이스를 설명할 수 있어야 한다.
- README 또는 sprint note에 선택한 구조, 선택 이유, 고려했지만 선택하지 않은 대안, 에러/예외 처리, 발표용 한 문장을 남긴다.
- 팀 싱크는 전체 코드 리뷰가 아니라 설계 판단, 막힌 부분, 구현 흐름, 발표 가능한 설명을 맞추는 시간으로 운영한다.
- 기술 스택은 React + Vite, FastAPI, PostgreSQL, pgvector를 기본 전제로 둔다.
- RAG, MCP, Agent는 게시판 흐름과 연결되어야 하며 별도 데모 버튼처럼 따로 놀지 않게 한다.

## Sprint 0. 스코프 고정

### 목표

프로젝트 범위를 고정하고, 팀 전체가 같은 방향으로 구현을 시작한다.

### 결정 사항

1. 모두 같은 서비스 컨셉으로 진행한다.
2. 모두 같은 AI 기능 시나리오로 진행한다.
3. 인증 방식은 JWT로 고정한다.
4. OAuth2, Session 심화 구현은 제외한다.
5. 각자 개인 repository에 구현한다.
6. 팀 싱크는 코드 전체 리뷰가 아니라 구현 흐름과 발표 설명 중심으로 진행한다.

### 기반 조건

새 회의 문서의 Sprint 1 기반 설계 항목은 여기 또는 Sprint 1 초반에 확인한다.

- React + Vite 앱이 실행된다.
- FastAPI 서버가 실행된다.
- PostgreSQL 연결이 준비된다.
- health check API가 동작한다.
- 최소 User/Post 모델이 있다.
- 게시글 목록 조회 또는 게시글 생성 API 중 최소 1개가 동작한다.
- FastAPI router 구조와 DB session 관리 방식이 정해져 있다.
- 이후 RAG를 위해 pgvector를 사용할 수 있는 PostgreSQL 기반 설계를 고려한다.
- 요청이 어떤 계층을 거쳐 DB까지 가는지 설명할 수 있다.

## Sprint 1. 인증 구현

### 목표

회원가입, 로그인, 현재 사용자 확인, 보호된 API를 구현한다.

### 구현 범위

- 회원가입
- 로그인
- JWT 발급
- 프론트엔드 token 저장
- 보호된 API 요청 시 token 첨부
- 백엔드에서 current_user 확인
- 인증 실패 처리
- 권한 실패 처리

### 기준

- 인증 방식은 JWT를 기본 구현 기준으로 한다.
- Session 방식은 이번 Sprint 1의 필수 구현 범위가 아니라 참고 또는 추후 확장 학습 항목으로 둔다.
- OAuth/OIDC는 이번 스프린트의 필수 구현 범위에서 제외한다.

### 완료 기준

- 회원가입이 가능하다.
- 로그인이 가능하다.
- 로그인 성공 시 JWT가 발급된다.
- 프론트엔드에 token이 저장된다.
- 게시글 작성 API 요청 시 token이 함께 전달된다.
- 백엔드에서 현재 사용자를 확인할 수 있다.
- 로그인하지 않은 사용자는 401 응답을 받는다.
- 다른 사용자의 글을 수정/삭제하려 하면 403 응답을 받는다.
- 선택한 JWT 흐름의 장점과 한계를 설명할 수 있다.
- 실패 케이스를 최소 1개 이상 확인한다.
- README 또는 sprint note에 인증 요청 흐름과 저장 위치를 남긴다.

### 팀 싱크 체크 포인트

- 토큰은 어디서 만들어지는가?
- 토큰에는 무엇이 들어가는가?
- 프론트엔드는 토큰을 어디에 저장하는가?
- 백엔드는 토큰을 어디서 검증하는가?
- 현재 사용자는 어떤 방식으로 API에 전달되는가?
- 401과 403은 어떻게 구분하는가?
- current_user는 router, service, repository 중 어디까지 전달되는가?
- XSS, token 탈취, refresh token 부재 같은 한계는 무엇인가?

## Sprint 3.5. AI 기능 스코핑

### 목표

RAG, MCP, Agent를 각각 따로 붙인 기능이 아니라 하나의 게시판 경험으로 연결되도록 AI 기능 시나리오를 확정한다.

이 단계는 별도 구현 스프린트로 길게 진행하지 않아도 된다. Sprint 4에 들어가기 전 설계 메모 또는 README 초안으로 남긴다.

### 결정할 것

- RAG 기능: 유사 게시글 추천, 중복 게시글 방지, 지식 Q&A 중 무엇을 우선할지 정한다.
- RAG 데이터: 게시글만 사용할지, 댓글/태그까지 포함할지 정한다.
- Embedding 저장 위치: posts 테이블 vector 컬럼 또는 post_embeddings 별도 테이블 중 선택한다.
- Embedding 생성 시점: 게시글 작성 시, 별도 버튼, 수동 재색인 중 선택한다.
- MCP 외부 서비스: GitHub, 외부 URL, 문서 API 등 하나 이상을 정한다.
- Agent 역할: 글쓰기 도우미, 태그 추천, 자료 조사 등 핵심 역할을 정한다.
- AI 결과 저장 정책: 일회성 표시인지, 사용자 확인 후 저장인지 정한다.

### 완료 기준

- RAG, MCP, Agent 기능이 하나의 사용자 흐름으로 연결된다.
- pgvector embedding 테이블 구조 초안을 정했다.
- 사용할 LLM Provider와 Embedding Model 후보를 정했다.
- MCP가 호출할 외부 서비스를 정했다.
- Agent가 사용할 tool 목록을 정했다.
- README에 들어갈 AI 아키텍처 초안을 작성했다.

## Sprint 4. RAG 구현

### 목표

게시판 데이터와 연결된 RAG 기능을 구현한다.

### 추천 기능

사용자가 글을 작성할 때 기존 게시글 중 유사한 글을 찾아 추천하고 요약한다.

### 구현 흐름

1. pgvector extension을 준비한다.
2. 게시글 저장 시 제목/본문을 embedding한다.
3. embedding 결과를 PostgreSQL의 vector 컬럼 또는 별도 embedding 테이블에 저장한다.
4. 글 작성 화면에서 유사 글 찾기 기능을 제공한다.
5. 입력 중인 제목/본문을 embedding한다.
6. pgvector similarity search로 유사 게시글을 검색한다.
7. 검색 결과 top 3개를 출력한다.
8. LLM이 검색 결과를 요약한다.
9. 프론트엔드에서 추천 결과를 보여준다.

### 완료 기준

- 게시글 제목/본문을 embedding한다.
- pgvector를 사용할 수 있는 DB 구조가 준비되어 있다.
- PostgreSQL에 embedding 결과를 저장한다.
- 새 글 작성 내용과 유사한 게시글을 검색한다.
- 유사 게시글 top 3개를 보여준다.
- LLM이 검색 결과를 요약한다.
- 프론트엔드에서 추천 결과를 확인할 수 있다.
- README에 RAG 구조를 설명할 수 있다.

### 제외 범위

- 고급 chunking
- 복잡한 reranking
- 고급 평가 지표
- 권한별 retrieval
- 실시간 streaming
- 고급 vector DB 튜닝

Sprint 4의 핵심은 RAG가 실제 게시판 데이터와 연결되어 작동하는 것이다.

### 팀 싱크 체크 포인트

- 무엇을 embedding했는가?
- embedding은 posts 테이블에 저장했는가, 별도 테이블에 저장했는가?
- vector 컬럼 차원 수는 어떻게 정했는가?
- metadata는 무엇을 저장했는가?
- 유사도 기준은 무엇인가?
- LLM prompt에는 어떤 context가 들어가는가?

## Sprint 5. MCP 구현

### 목표

외부 서비스를 호출하는 MCP Server를 구현하고 게시판 기능과 연결한다.

### 요구사항

- MCP Server 구현
- JSON-RPC 기반 요청/응답
- 외부 서비스 1개 이상 연동
- API Key 또는 권한 관리 전략 설명

### 추천 기능

URL 메타데이터 또는 본문 요약 가져오기

### 사용자 흐름

1. 글 작성 화면에서 URL을 입력한다.
2. MCP tool을 호출한다.
3. 외부 페이지 제목, 요약, 링크를 가져온다.
4. 결과를 게시글 내용에 참고자료로 삽입한다.

### 완료 기준

- MCP Server가 존재한다.
- JSON-RPC 요청/응답 형태가 구현되어 있다.
- 외부 서비스 하나를 실제 호출한다.
- API key 또는 권한 관리 방식을 README에 설명한다.
- 외부 호출 실패 시 에러 처리가 있다.
- 프론트엔드에서 결과를 확인할 수 있다.
- 게시판 기능 또는 Agent 기능 안에서 MCP 결과가 사용된다.

### 발표 설명 포인트

MCP는 LLM 또는 애플리케이션이 외부 시스템을 도구처럼 호출하기 위한 인터페이스다.
이번 프로젝트에서는 MCP Server에 외부 정보 조회 tool을 만들었다.
클라이언트 또는 백엔드가 JSON-RPC 형식으로 tool을 호출하면, MCP Server가 외부 API를 호출하고 결과를 반환한다.
이 결과를 게시글 작성에 활용한다.

### 팀 싱크 체크 포인트

- MCP tool 이름은 무엇인가?
- JSON-RPC request/response는 어떻게 생겼는가?
- 어떤 외부 서비스를 호출했는가?
- API Key는 어디에 저장했는가?
- 외부 호출 실패 시 사용자는 무엇을 보는가?
- MCP 결과는 게시글에 저장되는가, 일회성인가?
- Agent가 이 MCP tool을 사용할 수 있는 구조인가?

## Sprint 6. Agent 구현

### 목표

RAG와 MCP를 활용하는 글쓰기 도우미 Agent를 구현한다.

### 추천 기능

사용자가 주제를 입력하면 Agent가 유사 게시글을 찾고, 외부 자료를 가져오고, 게시글 초안과 태그를 제안한다.

### 사용자 흐름

사용자 입력 예시:

```text
JWT 인증 방식에 대한 글을 쓰고 싶어
```

Agent 실행 흐름:

1. RAG tool 호출
2. 기존 유사 게시글 검색
3. MCP tool 호출
4. 외부 자료 조회
5. LLM 호출
6. 게시글 초안 작성
7. 태그 추천
8. 사용자에게 결과 표시

### 완료 기준

- Agent가 tool을 호출한다.
- RAG tool 또는 MCP tool 중 최소 하나 이상을 실제 사용한다.
- 가능하면 RAG와 MCP를 모두 사용한다.
- Agent state에 사용자 입력, tool 결과, 최종 응답이 남는다.
- 최대 반복 횟수 제한이 있다.
- tool 실패 시 예외 처리가 있다.
- 프론트엔드에서 실행할 수 있다.
- 사용자 확인 없이 DB에 자동 저장하지 않는다.
- README에 Agent 구조를 설명할 수 있다.

### 발표 설명 포인트

Agent는 사용자의 요청을 받아 필요한 도구를 선택한다.
이번 구현에서는 RAG 검색 도구와 MCP 외부 정보 조회 도구를 tool로 제공했다.
Agent는 먼저 기존 게시글을 검색하고, 필요하면 외부 정보를 조회한 뒤, 그 결과를 종합해 게시글 초안과 태그를 생성한다.
무한 루프를 막기 위해 최대 실행 횟수를 제한했고, tool 호출 실패 시 사용자에게 에러 메시지를 반환하도록 했다.

- 무한 루프는 어떻게 막는가?
- tool 실패 시 어떻게 처리하는가?
