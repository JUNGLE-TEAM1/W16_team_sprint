# Sprint 로드맵: Sprint 1부터 Sprint 4까지

## Sprint 1. API 계약 + 데이터 흐름 + 기본 구조

이미 진행한 스프린트입니다.

목표:

- 요청 흐름
- 라우터/서비스/레포지토리
- 스키마/모델
- DB 세션
- 트랜잭션
- 에러 처리
- 기본 API 계약

완료 기준:

- 팀원 모두가 게시글 생성 요청 하나를 `프론트 -> 라우터 -> 서비스 -> 레포지토리 -> DB -> 응답` 흐름으로 설명할 수 있다.

## Sprint 2. 회원가입/로그인 + 인증/인가

기존 Sprint 2입니다. 그대로 진행하되 구현 완료형으로 바꿔야 합니다.

완료 기준:

- 회원가입/로그인이 된다.
- 로그인한 사용자만 글 작성/수정/삭제가 된다.
- 인증 실패/권한 실패를 구분한다.
- 프론트에서 로그인 상태를 사용할 수 있다.

## Sprint 3. 게시판 필수 기능 1: 게시글 CRUD + 댓글 + 태그

필수 추가 스프린트입니다.

완료 기준:

- 게시글 CRUD가 된다.
- 댓글 작성/조회/삭제가 된다.
- 태그 생성/연결/조회가 된다.
- Post, Comment, Tag 관계를 설명할 수 있다.
- 프론트 화면에서 사용할 수 있다.

중요한 학습 포인트는 DB 관계입니다.

- `User -> Post`
- `Post -> Comment`
- `Post -> Tag`
- `Tag -> Post`

특히 태그는 N:M 관계가 나올 수 있어서 ERD, join table, index 개념을 같이 볼 수 있습니다. PDF 로드맵에도 ERD, 1:1/1:N/N:M, PK/FK/Index, join, normalization 같은 데이터 모델링 항목이 포함되어 있습니다.

구현할 때는 기능별로 구조를 나눕니다.

```text
post_router/comment_router/tag_router
post_service/comment_service/tag_service
post_repository/comment_repository/tag_repository
post_schema/comment_schema/tag_schema
post_model/comment_model/tag_model
```

## Sprint 4. 게시판 필수 기능 2: 페이징 + 검색 + 목록 UX

필수 추가 스프린트입니다.

완료 기준:

- 게시글 목록이 페이지 단위로 조회된다.
- 검색어로 게시글을 찾을 수 있다.
- 태그 기반 필터링이 가능하면 더 좋다.
- 프론트에서 페이지 이동과 검색이 된다.
- 검색 결과 없음/로딩/에러 상태가 처리된다.

이 스프린트가 중요한 이유는 RAG와도 연결되기 때문입니다.

검색/페이징이 없는 게시판은 데이터 탐색 구조가 약합니다. RAG도 결국 "관련 데이터를 잘 찾는 기능"이기 때문에, 일반 검색과 AI 검색의 차이를 이해하려면 기본 검색 구조가 먼저 있어야 합니다.

## 공통 완료 기준

모든 기능 스프린트에는 프론트-백엔드 연결을 포함합니다.

> API만 끝나면 미완료. 화면에서 실제로 사용할 수 있어야 완료.

즉, 각 기능의 완료 기준은 항상 다음을 포함해야 합니다.

- 백엔드 API가 있다.
- DB 모델과 관계가 있다.
- 서비스/레포지토리 책임이 나뉘어 있다.
- 화면에서 실제로 호출하고 사용할 수 있다.
- loading/error/success 상태가 보인다.
- 에러가 사용자에게 표시된다.
