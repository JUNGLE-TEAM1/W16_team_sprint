- 프로젝트 구조
- 의존성 파일
- 실행 방법

- 이후 Sprint 3 화면 확장 방향

반대로 이번 Sprint 2를 백엔드 API 중심으로 끝내면 구현 범위는 작아지지만, `development-order.md`의 "최소 화면 구성"은 별도 Sprint 또는 별도 repository 작업으로 남는다.

## 5. 선택지

### A. 이번 Sprint 2는 백엔드 API만 완성한다.

범위:
- 게시글 CRUD 보강
- 댓글 생성/조회/수정/삭제 API
- 인증/권한/404 테스트
- 문서에 최소 화면은 후속 작업으로 남긴다.

의미:
- 현재 repository 구조와 맞다.
- 새 프론트엔드 의존성을 도입하지 않는다.
- Sprint 2의 API 완료 기준은 충족하지만, 최소 화면 구성은 미완료로 기록한다.

### B. 이번 repository에 최소 프론트엔드 앱을 새로 만든다.

범위:
- 새 프론트엔드 앱 추가
- 로그인 페이지
- 게시글 목록
- 게시글 작성
- 게시글 상세
- 댓글 입력/목록
- token 저장과 API 호출

의미:
- `development-order.md`의 최소 화면 구성까지 직접 다룬다.
- 새 빌드 도구와 패키지 의존성을 도입한다.
- 구현량과 검증 범위가 커진다.

### C. 프론트엔드는 별도 문서/흐름으로만 설계하고 구현하지 않는다.

범위:
- 백엔드 API 구현
- 화면/API 연결 방식 문서화
- 실제 프론트엔드 코드는 만들지 않는다.

의미:
- 발표나 학습 설명에는 도움이 된다.
- 실제 최소 화면 동작은 검증하지 않는다.
- A보다 문서가 많고, B보다 구현 범위가 작다.
