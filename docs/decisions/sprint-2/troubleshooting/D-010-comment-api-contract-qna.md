# D-010 Comment API Contract Q&A

Date: 2026-06-15
Sprint: 2
Decision: D-010. 댓글 API 계약은 어떻게 둘 것인가?

## Final Accepted Prompt

통과된 Decision:
- D-010. 댓글 API 계약은 어떻게 둘 것인가?

사용자 최종 답변:

```md
선택: A

이유: 형식 통일로 인한 유지 보수 편의성, 개발 편의성

댓글 수정 API 포함 여부: 댓글 수정도 포함한다.

댓글 조회 방식: nested API로 통일한다.

응답에 작성자 식별/표시 정보 노출 여부: 작성자 식별/표시 정보는 불필요하다. 다만 댓글 응답은 A안 그대로 `id`, `post_id`, `content`, `created_at`, `updated_at`을 반환한다.

테스트에서 확인할 것: 정상 처리, 404 처리, 비로그인 401, 타인 수정/삭제 403을 확인한다.
```

Codex 평가:
- Pass

Pass 이유:
- A 선택이 명확하다.
- nested URL로 생성/조회/수정/삭제를 통일한다.
- 댓글 수정 API 포함 여부가 명확하다.
- 작성자 식별/표시 정보를 응답에 노출하지 않는다는 기준이 D-009와 일관된다.
- API 테스트 범위가 정상/404/401/403으로 정리됐다.

보완 질문 여부:
- 있음.
- "comment id만 응답"의 의미가 작성자 정보 제외인지, 진짜 id-only 응답인지 확인했다.
- 인증/권한 실패 테스트 포함 여부를 확인했다.

아직 남은 후속 분기:
- 최소 화면 구현 범위
- 댓글 권한 검사 계층은 Level 2로 낮춰 service 계층 기본 선택 가능

최종 결론:
- 댓글 API는 게시글 하위 nested API로 구현한다.
- 응답에는 작성자 식별/표시 정보를 노출하지 않는다.
- 구현은 남은 필수 Level 3 후보 확인 뒤 시작한다.
