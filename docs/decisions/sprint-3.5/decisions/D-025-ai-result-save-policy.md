## 1. 현재 분기

D-018에서 AI 사용자 흐름은 글 작성 시 유사 게시글 추천과 중복 게시글 방지로 확정했다.

D-024에서 Agent는 RAG/MCP를 사용하는 글쓰기 도우미로 확정했다.

이제 AI가 만든 결과를 게시글에 어떻게 반영할지 정해야 한다.

## 2. 이번에 선택하지 않는 분기

이번 Decision은 AI 결과 저장 정책만 확정한다.

다음 항목은 이번 선택으로 자동 확정하지 않는다.

- 프론트엔드 미리보기 UI 상세
- 저장 버튼 문구
- Agent 결과 DB 저장 여부
- 외부 참고자료를 게시글에 포함할지 여부
- AI 생성 표시 문구
- 감사 로그 또는 사용 이력 저장 여부

## 3. 선택지

### A. AI 결과는 미리보기로만 반환하고, 사용자가 확인해야 작성 화면에 반영한다

Agent 결과는 바로 게시글로 저장하지 않는다. 사용자가 미리보기를 보고 반영 버튼을 누르면 작성 폼에 채워지고, 최종 게시글 저장은 기존 작성 흐름을 따른다.

장점:
- 사용자가 최종 내용을 통제한다.
- AI hallucination이나 부정확한 참고자료가 자동 저장되는 위험을 줄인다.
- 기존 게시글 작성 API와 권한 흐름을 크게 흔들지 않는다.
- 팀 싱크 입력의 “사용자가 확인해야만 게시글에 반영”과 맞다.

단점:
- 미리보기와 반영 UI가 필요하다.
- Agent 결과를 어디까지 폼에 채울지 UI 세부 결정이 필요하다.

### B. AI 결과를 임시 draft로 서버에 저장한다

Agent 결과를 별도 draft resource로 저장하고, 사용자가 나중에 이어서 편집한다.

장점:
- 사용자가 작업을 이어가기 좋다.
- Agent 결과 이력을 남길 수 있다.

단점:
- draft 모델과 API가 새로 필요하다.
- 저장 정책, 삭제 정책, 권한이 추가된다.
- Sprint 4 MVP 범위가 커진다.

### C. AI 결과를 바로 게시글로 저장한다

Agent가 생성한 초안을 곧바로 게시글 생성 API에 반영한다.

장점:
- 데모 흐름이 짧다.
- 구현 화면 단계가 적어 보일 수 있다.

단점:
- 사용자 확인 없이 AI 생성물이 저장된다.
- 잘못된 외부 참고자료나 부정확한 초안이 게시될 수 있다.
- 기존 게시글 작성 권한과 validation 흐름을 더 조심해야 한다.

## 4. Codex 추천

추천은 A다.

AI 결과는 보조 도구의 결과이며, 최종 게시글은 사용자가 확인하고 저장해야 한다.

이 방향은 D-018 팀 싱크 입력과 일치하고, Sprint 4 MVP에서 데이터 모델을 과도하게 늘리지 않는다.

## 7. 롤백 계획 초안

이 Decision 자체는 아직 구현을 변경하지 않는다.

구현 단계에서 바뀔 수 있는 범위:
- Agent 응답 schema
- 프론트엔드 미리보기/반영 UI
- 작성 폼 state
- 관련 테스트

되돌릴 때 확인할 범위:
- Agent 결과 반영 버튼
- 작성 폼 자동 채움 로직
- AI 미리보기 UI
- Agent 응답 schema

데이터 손실 가능성:
- 이 Decision 단계에서는 없음
- AI 결과를 서버에 저장하지 않는 선택이면 AI 미리보기 결과는 재요청 전까지 휘발될 수 있다.

원래 상태 보장:
- AI 기능 없이도 기존 게시글 작성 흐름이 동작해야 한다.
- Agent 실패가 기존 작성 폼 저장을 막지 않아야 한다.

재검토 조건:
- 팀이 AI draft 저장을 필수 요구사항으로 추가하는 경우
- 미리보기 결과를 발표 중 재사용해야 해서 서버 저장이 필요해지는 경우
- 사용자가 AI 결과 이력 확인 기능을 요구하는 경우
