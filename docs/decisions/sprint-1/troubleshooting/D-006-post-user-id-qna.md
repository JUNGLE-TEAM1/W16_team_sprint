# D-006 Q&A. 게시글 `user_id` 추가

Date: 2026-06-14
Related Decision: `docs/decisions/sprint-1/decisions/D-006-post-user-id.md`

## 1. 쟁점

D-006은 게시글에 `posts.user_id`를 추가할지 여부만 결정한다.

사용자는 "다음 분기로 뭘 남겨두냐, 지금 해결해야 한다"는 취지로 답했다. Codex는 이것을 Hold로 평가했다. 이유는 다음과 같다.

- 한 Decision은 하나의 핵심 분기만 통과시킨다.
- 다음 분기로 남긴다는 말은 미루자는 뜻이 아니다.
- `author_name` 유지 여부, 표시용 작성자 생성 방식, 응답 노출, 권한 계층은 D-006의 결과에 의존하지만 D-006으로 자동 확정되지 않는다.

## 2. 보완 질문

```md
1. D-006에서는 `posts.user_id` 추가만 확정하고, `author_name` 유지/생성 방식은 다음 후보에서 바로 이어서 결정해도 괜찮나요?
2. `posts.user_id`는 권한 판단용이고, `author_name`은 표시용인지 여부는 별도 결정으로 다루는 것에 동의하나요?
```

## 3. 사용자 보완 답변

```md
1. 좋아
2. 음 그 부분은 나중에 만들어 가야 하는 상황이고 지금 고민하는 책임이랑은 다른 부분이니 오케이
```

## 4. 결론

평가: Pass

통과 이유:
- `posts.user_id` 추가 여부만 D-006에서 확정하는 흐름을 수락했다.
- `posts.user_id`는 권한 판단용, `author_name`은 표시용 책임으로 별도 검토할 수 있음을 인정했다.

## 5. Final Accepted Prompt

```md
선택: A

이유: 게시글 API의 권한 확인에는 사용자를 안정적으로 식별할 값이 필요하다. 문자열 작성자 값이나 표시용 이메일은 권한 기준으로 쓰기 어렵다.

권한 검사는 어떤 값으로 할 것인가: `posts.user_id`와 현재 인증 사용자 `current_user.id`를 비교한다.

DB 변경 비용에 대한 생각: Sprint 1 이후에도 유지할 기본 구조이므로 `posts.user_id` 컬럼 추가 비용을 감수한다.

아직 다음 분기로 남겨둘 것: `author_name` 유지 여부, 생성 위치, 응답 노출, 권한 검사 계층은 D-006에서 자동 확정하지 않고 바로 이어지는 별도 분기에서 순차 결정한다.
```
