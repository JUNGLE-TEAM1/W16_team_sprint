# 학습 우선순위 정리

## 제 결론

가장 먼저 공부해야 할 주제는 **"API 계약 + 도메인/데이터 모델링 + 요청 흐름"**입니다.

단순히 말하면,

> **프론트에서 어떤 요청이 들어오고, 백엔드가 어떻게 처리하고, DB에 어떤 데이터가 저장되고, 어떤 응답과 에러가 돌아가는지 팀 전체가 같은 그림으로 설명할 수 있게 만드는 것**이 첫 번째입니다.

그다음 우선순위는 이렇게 두는 것을 추천합니다.

1. **API 계약 + 도메인/데이터 모델링 + 요청 흐름**
2. **인증/인가 + 보안 기본**
3. **프로젝트 구조/아키텍처 + 에러 처리/로깅/설정**
4. **프론트엔드-백엔드 연결 방식 + 상태 관리**
5. **AI/RAG/LLM 기능 흐름 - 프로젝트가 AI 중심이면 2번으로 올리기**
6. **비동기 처리/캐싱/실시간/신뢰성 - 필요한 경우에만 깊게**

PDF에 나온 빨간색 항목들은 좋은 후보군이지만, 전부 같은 중요도를 가진 것은 아닙니다. PDF에는 프론트엔드, 백엔드, API, 인증, 보안, 데이터 레이어, 아키텍처, 로깅, 테스트, 설정 등 굉장히 넓은 범위의 풀스택 로드맵이 들어 있습니다. 이걸 "다 공부해야 할 목록"으로 보면 실패하고, **나만무에서 의사결정을 빠르게 하기 위한 공통 언어 후보군**으로 봐야 합니다.

---

# 1. 기준은 이렇게 세워야 합니다

주제를 고를 때 기준은 "중요해 보이는가?"가 아니어야 합니다.

거의 모든 주제가 중요해 보입니다.
JWT도 중요하고, React 상태 관리도 중요하고, DB도 중요하고, 아키텍처도 중요하고, 테스트도 중요합니다. 그래서 "중요도"만 보면 결정을 못 합니다.

대신 기준은 이것이어야 합니다.

> **나만무에 들어갔을 때, 지금 몰라서 가장 큰 비용을 치르게 되는가?**

조금 더 구체적으로는 아래 기준입니다.

---

## 기준 1. 나중에 알면 늦는가?

이게 가장 중요합니다.

구현하면서 자연스럽게 배울 수 있는 것은 뒤로 미뤄도 됩니다.
하지만 초반에 팀 전체가 같은 기준을 갖고 있지 않으면 나중에 구조를 뒤집어야 하는 것은 먼저 해야 합니다.

예를 들어 React에서 특정 훅을 어떻게 쓰는지는 나중에 구현하면서 배워도 됩니다.
하지만 API 응답 형식, DB 관계, 인증 흐름, 백엔드 레이어 구조는 나중에 바꾸기 어렵습니다.

그러면 첫 번째 우선순위는 자연스럽게 **API, 데이터, 인증, 아키텍처** 쪽으로 갑니다.

---

## 기준 2. 여러 파트가 동시에 영향을 받는가?

한 사람의 담당 기능 안에서만 끝나는 지식은 우선순위가 낮습니다.
여러 파트가 동시에 영향을 받는 지식은 우선순위가 높습니다.

예를 들어 API 설계는 프론트, 백엔드, DB, 인증, AI 기능 모두에 영향을 줍니다.
인증도 프론트 라우팅, 백엔드 미들웨어, DB 유저 테이블, 보안 정책에 모두 영향을 줍니다.
아키텍처도 모든 백엔드 코드의 위치와 책임을 결정합니다.

반대로 특정 UI 프레임워크의 세부 사용법이나 특정 라이브러리 API는 담당자가 깊게 파도 됩니다.

즉, 지금 팀 학습에서 우선해야 할 것은 **개별 구현 지식**이 아니라 **접점 지식**입니다.

---

## 기준 3. 팀원이 같은 말을 못 알아들으면 의사결정이 느려지는가?

이번 논의의 핵심은 "모두가 모든 기능을 깊게 알아야 하느냐"가 아니었습니다.

정답은 아닙니다.
모두가 같은 깊이로 알 필요는 없습니다.

하지만 모두가 **같은 말을 알아들을 정도**는 되어야 합니다.

예를 들어 한 사람이 RAG를 맡았다고 해도, 다른 팀원들이 RAG, 임베딩, 검색, 컨텍스트 주입, 응답 생성 흐름을 아예 모르면 회의가 안 됩니다. 담당자가 "임베딩 방식 때문에 검색 품질이 떨어진다"고 말했는데 다른 사람들이 그 말 자체를 이해하지 못하면 팀 의사결정이 막힙니다.

따라서 팀 학습은 깊이를 같게 만드는 것이 아니라, **대화 가능한 최소 공통선을 맞추는 것**이어야 합니다.

---

## 기준 4. 실패했을 때 치명적인가?

실패해도 금방 고칠 수 있는 것은 뒤로 미뤄도 됩니다.
실패하면 보안, 데이터, 일정, 구조 전체가 흔들리는 것은 먼저 해야 합니다.

치명도가 높은 주제는 보통 이런 것들입니다.

* 인증/인가
* DB 모델링
* API 계약
* 에러 처리
* 데이터 무결성
* 보안
* 프로젝트 구조
* AI 핵심 기능의 책임 범위
* 비동기 처리나 외부 API 호출 실패 대응

예쁜 UI, 세부 CSS, 특정 라이브러리 문법은 중요하지만 지금 최우선은 아닙니다.

---

## 기준 5. 1~2시간 스프린트로 공통 이해를 만들 수 있는가?

지금 팀이 하려는 방식은 짧게 공부하고, 짧게 싱크를 맞추는 방식이었습니다. 이전 논의에서도 발표식 학습보다는 각자 공부하고 핵심 요점과 질문을 공유하는 방식으로 방향이 잡혔습니다.

그렇다면 너무 큰 주제는 안 됩니다.

"백엔드 아키텍처 완전 정복"은 안 됩니다.
"인증 완전 정복"도 안 됩니다.
"데이터베이스 완전 정복"도 안 됩니다.

대신 이렇게 쪼개야 합니다.

> "게시글 작성 요청 하나가 프론트 -> API -> 서비스 로직 -> DB -> 응답까지 어떻게 흐르는가?"

또는

> "로그인하면 access token이 언제 발급되고, 이후 요청에서 백엔드는 어떻게 사용자를 확인하는가?"

이런 식으로 **작고 명확한 흐름 단위**로 공부해야 합니다.

---

# 2. 그래서 첫 번째 주제는 왜 API + 데이터 흐름인가?

저는 첫 번째 주제를 **인증/JWT**가 아니라 **API 계약 + 도메인/데이터 모델링 + 요청 흐름**으로 잡는 것을 추천합니다.

이유는 명확합니다.

인증도 결국 API를 통해 들어옵니다.
프론트 상태 관리도 결국 API 응답을 받아서 합니다.
DB 설계도 API가 다루는 리소스와 연결됩니다.
AI 기능도 결국 입력, 처리, 저장, 응답의 흐름을 갖습니다.
아키텍처도 "요청이 들어왔을 때 어느 레이어가 무엇을 책임지는가"를 정하는 문제입니다.

즉, **API와 데이터 흐름은 거의 모든 주제의 뼈대**입니다.

PDF에도 API 영역에는 REST, gRPC, GraphQL, API design, HTTP 3xx/4xx/5xx 에러 처리 등이 있고, 데이터 레이어에는 RDBMS, SQL, ERD, 정규화, PK/FK/Index, transaction, lock/deadlock 등이 포함되어 있습니다. 이 둘은 따로 떨어진 지식이라기보다, 실제 서비스에서는 항상 같이 움직입니다.

첫 주제는 이렇게 잡으면 좋습니다.

> **"우리 서비스에서 하나의 핵심 기능이 실행될 때, 프론트 요청부터 DB 저장과 응답까지 어떤 흐름으로 지나가는가?"**

예시는 게시판 기준이면 충분합니다.

* 게시글 작성
* 게시글 목록 조회
* 댓글 작성
* 로그인 후 내 글 수정
* AI 요약 요청 후 결과 저장

이런 흐름 하나를 잡고 보면 됩니다.

# 3. 추천 우선순위

## 1순위. API 계약 + 도메인/데이터 모델링 + 요청 흐름

### 왜 먼저인가?

이건 모든 협업의 접점입니다.

프론트는 API 명세를 보고 개발합니다.
백엔드는 API 명세를 기준으로 라우터와 서비스를 나눕니다.
DB는 API에서 다룰 리소스를 저장할 구조를 가져야 합니다.
AI 기능도 입력과 출력이 API와 데이터 구조에 묶입니다.

이게 안 맞으면 팀원들이 각자 다른 제품을 만들게 됩니다.

예를 들어 한 사람은 `userId`가 항상 있다고 생각하고, 다른 사람은 비로그인 사용자도 가능하다고 생각하면 설계가 갈라집니다.
한 사람은 게시글 삭제가 실제 삭제라고 생각하고, 다른 사람은 soft delete라고 생각하면 DB와 API가 달라집니다.
한 사람은 AI 응답을 매번 새로 생성한다고 생각하고, 다른 사람은 DB에 저장한다고 생각하면 아키텍처가 달라집니다.

이 차이는 나중에 코드 짜다가 발견하면 비용이 큽니다.

### 공부할 내용

* REST API 기본
* HTTP method
* HTTP status code
* request/response schema
* validation
* error response 형식
* ERD
* 1:1, 1:N, N:M
* PK, FK, index
* transaction이 필요한 순간
* 프론트에서 loading/error/success를 어떻게 다루는지

### 싱크 목표

이 주제의 목표는 "REST를 잘 안다"가 아닙니다.

목표는 이것입니다.

> **팀원 모두가 하나의 기능에 대해 API, DB, 화면 흐름을 같이 그릴 수 있다.**

### 산출물

이 스프린트가 끝나면 아래가 남아야 합니다.

```text
1. 예시 기능 하나의 요청 흐름
2. API endpoint 초안
3. request/response 예시
4. error response 형식
5. 필요한 DB 테이블과 관계
6. 아직 결정하지 못한 질문 목록
```

이게 나오면 성공입니다.

---

## 2순위. 인증/인가 + 보안 기본

### 왜 두 번째인가?

인증은 거의 모든 서비스에서 빠르게 등장합니다.

그리고 인증은 나중에 "그냥 붙이면 되는 기능"이 아닙니다.
프론트 라우팅, 백엔드 미들웨어, DB 유저 테이블, 토큰 저장 방식, API 권한 처리, 보안 정책이 모두 연결됩니다.

회의에서도 로그인 공부 중 토큰을 놓칠 수 있다는 사례가 나왔습니다. 혼자 AI와 구현하면 "작동은 하는데 왜 그렇게 해야 하는지 모르는 코드"가 나오기 쉽고, 팀원들이 서로 질문하면서 핵심 누락을 잡아줄 수 있다는 이야기가 있었습니다.

PDF에서도 인증 영역에는 Session, JWT, OAuth2, OIDC가 있고, 보안 영역에는 HTTPS, OWASP Top 10, Rate limit, CSRF/CORS가 포함되어 있습니다.

### 공부할 내용

깊게 들어가기보다 아래를 먼저 맞추면 됩니다.

* 인증과 인가의 차이
* Session 방식
* JWT 방식
* access token / refresh token
* OAuth2와 OIDC가 필요한 상황
* 프론트에서 토큰을 어디에 둘지
* 백엔드에서 인증 사용자를 어떻게 확인할지
* CORS와 CSRF의 차이
* HTTPS가 왜 필요한지
* rate limit이 왜 필요한지
* "로그인된 사용자만 가능한 API"를 어떻게 구분할지

### 싱크 목표

> **로그인 요청부터 인증이 필요한 API 호출까지의 흐름을 팀원 모두가 설명할 수 있다.**

### 산출물

```text
1. 우리 팀 기본 인증 흐름 그림
2. Session/JWT/OAuth 중 언제 무엇을 쓸지 기준
3. access token / refresh token 역할 정리
4. 인증 필요한 API 표시 방식
5. 보안상 조심해야 할 기본 체크리스트
```

여기서 중요한 것은 "JWT 코드를 완벽히 구현하기"가 아닙니다.

**JWT가 왜 필요한지, 어디서 위험한지, 어떤 선택지가 있는지**를 팀 전체가 알아야 합니다.

## 3순위. 프로젝트 구조/아키텍처 + 에러 처리/로깅/설정

### 왜 세 번째인가?

아키텍처는 너무 일찍 공부하면 추상적입니다.
하지만 너무 늦게 정하면 코드가 중구난방이 됩니다.

그래서 API와 데이터 흐름을 먼저 본 뒤, 바로 프로젝트 구조를 맞추는 게 좋습니다.

PDF에도 MVC, Layered architecture, Modular monolith, clean architecture, hexagonal architecture, ADR, logging, testing, configuration 같은 항목들이 나옵니다.

하지만 여기서 clean architecture나 hexagonal architecture를 깊게 파는 것은 비추천입니다. 지금 필요한 건 고급 이론이 아니라, 팀이 당장 쓸 **최소 구조**입니다.

### 공부할 내용

* MVC가 무엇인지
* Layered architecture가 무엇인지
* router / controller / service / repository 역할
* FastAPI 기준 파일 구조
* React 기준 폴더 구조
* schema / DTO / model 차이
* error handling 위치
* logging을 어디서 할지
* config/env를 어떻게 관리할지
* ADR을 왜 쓰는지
* 테스트는 최소 어디까지 할지

### 싱크 목표

> **AI가 코드를 짜주더라도, 팀원들이 어느 파일에 무엇을 둬야 하는지 같은 기준을 가진다.**

이게 중요합니다.

AI를 쓰면 코드 생성은 빨라지지만, 구조가 없으면 각자 다른 방식으로 생성된 코드가 섞입니다. 그러면 나중에 합칠 때 힘들어집니다.

### 산출물

```text
1. 백엔드 기본 폴더 구조
2. 프론트엔드 기본 폴더 구조
3. 에러 응답 공통 형식
4. logging 최소 규칙
5. config/env 관리 규칙
6. ADR 작성 템플릿
```

---

## 4순위. 프론트엔드-백엔드 연결 방식 + 상태 관리

### 왜 네 번째인가?

프론트엔드 자체의 모든 내용을 지금 깊게 팔 필요는 없습니다.

React 컴포넌트, props, event, routing, 상태 관리는 중요합니다. PDF에도 프론트엔드 영역에 React, 컴포넌트 설계, 상태/Props, 이벤트, 라이프 사이클, 라우팅, 상태관리, 테스트, UI 프레임워크 등이 포함되어 있습니다.

하지만 지금 우선해야 하는 것은 "React를 잘 쓰는 법" 전체가 아닙니다.

지금 팀 차원에서 중요한 것은 이것입니다.

> **서버 데이터를 프론트에서 어떻게 가져오고, loading/error/success 상태를 어떻게 다루고, API 변경이 프론트에 어떤 영향을 주는가?**

즉, 프론트 학습도 API와 연결된 부분을 먼저 봐야 합니다.

### 공부할 내용

* server state와 client state의 차이
* API 호출 위치
* loading/error/success 처리
* form validation
* 라우팅과 인증 상태 연결
* protected route
* 전역 상태가 필요한 경우와 아닌 경우
* Zustand/Redux 같은 상태관리 도구가 필요한 기준
* 컴포넌트 분리 기준

### 싱크 목표

> **API가 주어졌을 때 프론트가 어떤 상태와 화면 흐름으로 연결되는지 팀원들이 이해한다.**

### 산출물

```text
1. API 호출 패턴 예시
2. loading/error/success 처리 규칙
3. 인증 상태 처리 방식
4. 상태관리 도구를 쓸지 말지 기준
5. 컴포넌트 분리 기준
```

---

## 5순위. AI/RAG/LLM 기능 흐름

이건 PDF에는 직접적으로 나오지 않지만, 여러분의 프로젝트가 AI 중심이라면 매우 중요합니다.

만약 나만무에서 AI 기능이 핵심이라면 이 주제는 **5순위가 아니라 2순위**로 올려야 합니다.

### 왜 조건부인가?

AI 기능은 프로젝트에 따라 중요도가 완전히 달라집니다.

만약 프로젝트의 핵심 가치가 AI 추천, AI 요약, RAG 검색, 에이전트 기능이라면 이걸 모르면 제품의 중심을 모르는 것입니다.
반대로 AI가 부가 기능이라면 초반에 너무 깊게 파는 것은 시간 낭비일 수 있습니다.

그래서 기준은 이것입니다.

> **AI 기능이 제품의 핵심 가치인가, 아니면 보조 기능인가?**

핵심 가치라면 API/데이터 흐름 다음에 바로 공부하세요.

### 공부할 내용

* LLM 요청/응답 흐름
* prompt 입력과 출력 구조
* RAG가 필요한 상황
* embedding이 무엇인지
* vector search가 무엇인지
* 검색된 context가 LLM에 어떻게 들어가는지
* AI 응답을 DB에 저장할지 말지
* 실패했을 때 fallback
* 비용/속도/품질 trade-off
* 비동기 처리 필요 여부
* 로그를 어떻게 남길지
* 평가를 어떻게 할지

### 싱크 목표

> **AI 기능을 "마법 상자"가 아니라 입력, 처리, 저장, 응답, 실패 대응이 있는 하나의 시스템으로 이해한다.**

### 산출물

```text
1. AI 기능 입력/출력 정의
2. RAG가 필요한지 여부
3. 저장할 데이터와 저장하지 않을 데이터
4. 실패 시 사용자 경험
5. 비용/속도/품질 trade-off
6. 담당자가 깊게 팔 영역과 팀 전체가 알아야 할 영역 구분
```

---

## 6순위. 비동기 처리/캐싱/실시간/신뢰성

이건 중요하지만, 대부분은 조건부입니다.

PDF에는 WebSocket, SSE, Job Queue, Redis, RabbitMQ, Kafka, retry, timeout, circuit breaker, idempotency 같은 항목들이 포함되어 있습니다.

하지만 지금 전부 깊게 공부할 필요는 없습니다.

### 먼저 공부해야 하는 경우

아래에 해당하면 우선순위를 올리세요.

* AI 응답 시간이 길다.
* 파일 업로드/분석이 있다.
* 백그라운드 작업이 필요하다.
* 실시간 알림이나 채팅이 핵심이다.
* 외부 API 호출이 많다.
* 같은 요청이 중복 실행되면 문제가 생긴다.
* 결과를 캐싱해야 비용이 줄어든다.

이런 경우가 아니라면 지금은 개념만 알면 됩니다.

### 공부할 내용

* 동기 처리와 비동기 처리 차이
* Job Queue가 필요한 상황
* Redis cache가 필요한 상황
* WebSocket과 SSE 차이
* timeout
* retry
* idempotency
* 실패한 작업을 어떻게 재처리할지

### 싱크 목표

> **느리거나 실패할 수 있는 작업을 API 요청 안에서 바로 처리할지, 백그라운드로 뺄지 판단할 수 있다.**

**지금 팀 싱크의 최우선 주제는 아니라는 뜻**입니다.

특히 Microservices, CQRS, Kafka, Sharding 같은 것은 **문제가 생긴 뒤에 필요한 해결책**에 가깝습니다. 아직 어떤 문제가 생길지도 모르는 상태에서 먼저 깊게 공부하면 추상 지식만 늘고 실제 나만무 준비에는 덜 도움이 됩니다.

# 5. 제가 추천하는 실제 학습 순서

여러분의 방식이 1~2시간 공부하고 30분~1시간 싱크하는 구조라면, 이렇게 가면 좋습니다.

# 6. 각 주제의 "완료 기준"을 정해야 합니다

그냥 공부하고 끝내면 안 됩니다.
각 스프린트는 완료 기준이 있어야 합니다.

저는 완료 기준을 이렇게 잡는 것을 추천합니다.

## 한 주제의 완료 기준

팀원들이 아래 질문에 답할 수 있으면 완료입니다.

```text
1. 이 개념은 무엇인가?
2. 왜 필요한가?
3. 어떤 선택지가 있는가?
4. 선택지별 장단점은 무엇인가?
5. 우리 프로젝트에서는 언제 필요할 가능성이 높은가?
6. 프론트/백엔드/DB/AI 중 어디에 영향을 주는가?
7. 나중에 더 깊게 팔 담당자는 누구인가?
8. 지금 팀 차원에서 합의할 기본값은 무엇인가?
```

이게 중요합니다.

팀 학습의 목표는 "모두가 같은 양을 공부했다"가 아닙니다.
목표는 **팀 의사결정 속도를 올리는 것**입니다.

그래서 매 스프린트가 끝날 때 최소한 하나는 남아야 합니다.

* 기본 선택
* 체크리스트
* 질문 목록
* ADR 초안
* 예시 흐름도
* API/DB 초안

아무 산출물 없이 "좋은 얘기 했다"로 끝나면 싱크가 맞았는지 확인할 수 없습니다.

# 7. 주제 선정 점수표

회의 때 바로 쓸 수 있게 기준표로 만들면 이렇게 됩니다.

각 주제를 0~3점으로 평가하세요.

| 기준 | 질문 |
| --- | --- |
| 선행성 | 이걸 알아야 다른 결정을 할 수 있는가? |
| 복구 비용 | 나중에 바꾸면 구조를 많이 고쳐야 하는가? |
| 접점성 | 프론트/백엔드/DB/AI 여러 파트가 동시에 영향을 받는가? |
| 실패 비용 | 잘못하면 보안, 데이터, 일정, 품질에 큰 문제가 생기는가? |
| 출현 확률 | 나만무에서 실제로 쓸 가능성이 높은가? |
| 지식 격차 | 팀원 간 이해 차이가 클 가능성이 높은가? |
| 스프린트 적합성 | 1~2시간 공부 후 싱크 가능한가? |

이 기준으로 보면 대략 이렇게 나옵니다.

| 주제 | 우선순위 | 이유 |
| --- | ---: | --- |
| API 계약 + 데이터 흐름 | 최상 | 모든 기능의 공통 접점이고, 나중에 바꾸기 어렵다 |
| 인증/인가 + 보안 | 최상 | 대부분 서비스에 필요하고, 실패 비용이 크다 |
| 아키텍처/프로젝트 구조 | 상 | AI 코드 생성이 많을수록 구조 기준이 중요하다 |
| AI/RAG/LLM 흐름 | 조건부 최상 | 프로젝트 핵심이면 제품 성공을 좌우한다 |
| 프론트-백엔드 상태 연결 | 상 | 화면과 API 연결에서 혼란이 자주 생긴다 |
| 비동기/캐싱/실시간 | 조건부 상 | AI, 알림, 긴 작업이 있으면 중요하다 |
| 테스트/로깅/config | 중상 | 초반에 최소 규칙은 필요하지만 깊은 학습은 나중에 가능하다 |
| UI 프레임워크 세부 | 중하 | 필요할 때 담당자가 빠르게 학습 가능하다 |
| Microservices/CQRS/Kafka/Sharding | 하 | MVP 초반에는 과한 경우가 많다 |

# 8. 최종 추천안

제가 팀장이라면 이렇게 결정하겠습니다.

> **남은 준비 기간의 학습 주제는 "나중에 의사결정을 늦추는 공통 접점"부터 한다.**
> 첫 주제는 API 계약과 데이터 흐름이다.
> 그다음 인증/보안, 프로젝트 구조, 프론트-백엔드 연결, AI 기능 흐름 순서로 간다.
> 단, 최종 프로젝트가 AI 중심이라면 AI 기능 흐름은 인증보다 먼저 또는 같은 우선순위로 올린다.

가장 중요한 것은 이겁니다.

**지금은 지식을 많이 쌓는 시간이 아니라, 팀의 의사결정 언어를 맞추는 시간입니다.**

그래서 첫 번째 질문은 "무엇을 공부할까?"가 아니라,

> **"나만무 때 회의가 막히지 않으려면, 모두가 어떤 말을 알아들어야 하는가?"**

여야 합니다.

그 기준으로 보면 첫 학습 주제는 **API + 데이터 흐름**이 가장 안전합니다.
이걸 먼저 맞추면 이후 인증, AI, 프론트 상태, 아키텍처를 모두 같은 그림 위에 얹을 수 있습니다.
